## SA-04 — Cache key now folds in the response-affecting mission metadata (fixed)

`src/upstreamwx/api/cache.py`

The response cache key (`mission_cache_key`) omitted `name` / `party_size` / `route_note`, while
the cached `GeneratedBriefing` embeds the request's `Mission` and the rendered briefing prints
`mission.name` (`render.py:114`, `structured.py:440`). Two users at the same place / activity /
window / radii therefore **collided on one entry**, and the second was served the **first's mission
name and presentation** — a cross-user disclosure and a cache-poisoning vector (pre-seed a
predictable location/window with a misleading label).

- The key now folds in the user-supplied metadata as a single collision-safe
  `repr((name, party_size, route_note))` part, so a request differing only in these fields gets a
  **distinct entry** and renders its **own** name. (`name` is the only field rendered today;
  `party_size`/`route_note` are included defensively so a future change that surfaces them cannot
  reintroduce the leak. Their *removal* — the audit's secondary suggestion — is left to SA-10
  privacy minimization, since they are still plumbed through the CLI and the PWA planner.)
- This is the audit's **"immediate fix."** The stronger conditions/presentation cache split (which
  would preserve cross-metadata cache *sharing*) is recorded as a deferred enhancement in the
  workplan §6; it is disproportionate here because SA-02 already bounds these fields, so the only
  cost — two *distinct* users briefing the identical conditions with different names no longer share
  an entry — is negligible (a single user reopening *their* mission keeps a stable name → still a
  hit → "reopening is free" preserved).

**Acceptance (all three met):** requests differing only in name/party_size/route_note never return
each other's metadata (`test_api_cache_isolation.py`); a conditions-only re-request still hits and a
metadata change misses and renders the current request; the "name is not part of identity" assertion
is replaced with its inverse.

## SA-03 — Scheduled refresh can no longer become unbounded, multi-day, or interactive-starving work (fixed)

`src/upstreamwx/api/service.py`, `scheduler.py`, `app.py`, `config.py`

`refresh_active` re-ingested **every** registered in-range mission on **every** 6-hourly cycle until
its window ended — so one request (window up to 10 days out, span up to 7 days) could pin ~2 weeks of
recurring ingest — and the `_active` registry was mutated from request threads and the scheduler
thread with **no lock**. Four controls were added, each independent of the access gate:

- **Registry lock** (rec 6). All `_active` reads/mutations — registration, the new "touch on view",
  eviction, pruning, the refresh snapshot — take one `threading.Lock`. The slow `generate_briefing`
  work runs **outside** the lock (snapshot under lock → generate → cache), so a long pass never
  blocks request-thread registration, and the old lock-free `list(self._active.items())` + concurrent
  `del`/`min(...)` races can no longer raise.
- **Refresh only recently-viewed missions** (rec 3 — the biggest lever). Each entry carries a
  `last_seen`, stamped at registration and **bumped on every `get_briefing` (hit or miss)** — a hit
  means the user reopened the app. A refresh regeneration does **not** bump it, so a fire-and-forget
  request stops refreshing after `api_active_refresh_ttl_s` (default **12 h ≈ two cycles**): one
  anonymous request becomes at most ~2 cycles of work, **not days**. An actively-planned mission stays
  warm.
- **Hard per-pass budget** (rec 5). A pass stops cleanly at `api_refresh_pass_max_items` (default 64)
  or `api_refresh_pass_max_seconds` (default 240 s); missions not reached refresh next cycle or on
  demand (NFR-6). Total work per pass is bounded regardless of registry size.
- **Yield to interactive work** (rec 6). Each regeneration acquires the **same `_gen_sem`** the request
  path uses (so scheduled + interactive gens never jointly exceed the concurrency cap) but waits only
  `api_refresh_gen_wait_s` (default 0.5 s) for a slot; if the host is busy serving real users the pass
  **defers** the rest. Refresh is strictly lower priority than interactive briefings.
- **Metrics** (rec 7). `refresh_active` records a `RefreshStats` (registry size, regenerated,
  pruned-ended, pruned-stale, deferred, skipped-by-budget, duration); the scheduler logs it each cycle
  and `/v1/health` echoes the last pass's counts and the new limits (counts only — SA-12-safe).

`refresh_active` still returns the regenerated count (an `int`), so existing scheduler/cycle tests are
unchanged. On rec 2 ("global quota substantially below 256") and rec 4 (dedup by cycle/domain): with
the recently-viewed TTL pruning idle missions and the per-pass budget capping work, the 256
registry-count cap is now a **memory ceiling, not the work bound**; cross-mission domain dedup depends
on the deferred SA-04 conditions split (workplan §6), and the GEFS/REFS grid cache (per cycle) and
watershed cache (per point) already dedupe the expensive sub-steps across a pass.

**Acceptance (all three met):** creating many missions cannot cause unbounded/multi-day work (TTL
prune + item budget, `test_api_scheduler_budget.py`); a pass stops within budget and defers instead of
starving interactive briefings (busy-slot test); concurrent register/touch/evict/refresh raise nothing
and leave a consistent registry (lock test).

## New configuration (all default to safe values; documented in `deploy/upstreamwx.env.example`)

| Setting | Default | Purpose |
|---|---|---|
| `api_active_refresh_ttl_s` | `43200` (12 h) | refresh only missions viewed within this window (0 disables) |
| `api_refresh_pass_max_items` | `64` | per-pass regeneration cap (0 = unlimited) |
| `api_refresh_pass_max_seconds` | `240` | per-pass wall-clock budget (0 = unlimited) |
| `api_refresh_gen_wait_s` | `0.5` | how long refresh yields a slot to interactive work |

`/v1/health.limits` echoes all four, plus a new top-level `refresh` block with the last pass's counts.
SA-04 adds no configuration (a cache-key change only).

## Not addressed here (separate findings / deferred, per the workplan)

- Conditions/presentation cache split (SA-04 stronger design) and cross-mission domain dedup (SA-03
  rec 4) — larger refactor, recorded in workplan §6.
- Removing `party_size`/`route_note` if they stay unused — SA-10 privacy minimization.
- Shared-store counters/registry for a multi-worker host — the M0.1.1 upgrade the cache already
  documents; in-process is correct for the single-worker deployment.
